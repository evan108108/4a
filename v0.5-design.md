# 4A v0.5 — Private Audiences, Deterministic Keys, Pasted Invites, Federated Directory

**Status:** Design draft (2026-04-28; §2.5 corrected 2026-04-28 to align with the email-thread decision that everyone — including ChatGPT/Claude.ai users — can use private mode via capability-based delegation). Successor to the [v0 spec](./SPEC.md). Targets the first release where two humans can create a private group, invite each other, and exchange knowledge end-to-end on top of Nostr.
**Pronunciation:** "four-A, point five."
**Scope:** Five things. Audiences (private group addressing). Key management (the OAuth-derived custodial scheme already shipped in Phase 2). Invites (a single paste field that produces or consumes a claimable link). Directory (federated NIP-05 lookup with client-side petnames). A worked example that walks the whole stack.

This document is a design — it locks decisions and shows how the pieces fit. Normative event shapes will follow in a Phase 3-style spec stub once the design is reviewed. Where v0 already specifies something (event envelope, BLAKE3 tagging, JSON-LD context, OAuth derivation) this document links and does not repeat.

Background reading:
- [`SPEC.md`](./SPEC.md) — v0 normative spec (event envelope, public kinds, identity paths)
- [`ARCHITECTURE.md`](./ARCHITECTURE.md) — Phase 1/2 deployment, KMS-backed derivation
- [`kind-assignments.md`](./kind-assignments.md) — kind block, including reserved encrypted variants 30510–30514
- [`credibility-attestations.md`](./credibility-attestations.md), [`credibility-graphs.md`](./credibility-graphs.md), [`credibility-sybil.md`](./credibility-sybil.md) — orthogonal credibility design

## 1. Audiences

### 1.1 What an audience is

An **audience** is a named recipient set — `team-design`, `enginable-eng`, `evan-and-allison` — that a publisher can address as if it were a single recipient. v0.5 introduces audiences so two or more people can post 4A objects (Observation, Claim, Entity, Relation) to each other without those events being world-readable.

An audience is *not* a Nostr group in the [NIP-29](https://github.com/nostr-protocol/nips/blob/master/29.md) sense. NIP-29 puts groups inside a relay's namespace, with the relay as authority. We want audiences to be relay-portable: the same audience can fan out across `relay.damus.io`, `nos.lol`, a private relay, and a future `relay.4a4.ai` without permission from any one operator. So we model an audience as **a per-epoch keypair plus a roster of member pubkeys**, and we lean on existing Nostr encryption primitives ([NIP-44 v2](https://github.com/nostr-protocol/nips/blob/master/44.md) for confidentiality, [NIP-17 gift-wrap](https://github.com/nostr-protocol/nips/blob/master/17.md) for metadata privacy) to address it.

### 1.2 The two keys at play

For every audience there are two distinct kinds of keypair:

1. **Audience identity key** (`aud_id`). A long-lived secp256k1 keypair representing the audience as an entity. Its pubkey is what users see when picking a roster (`@team-design`). Its private key signs the audience's own declaration event and any audience-level metadata changes (name, description, public roster). It does **not** decrypt messages.
2. **Audience epoch key** (`aud_epoch_n`). A separate secp256k1 keypair generated per epoch. The epoch private key is what members hold; addressing a message to the audience means encrypting it (via NIP-44) to the **current epoch's pubkey**. Decryption requires knowing the epoch private key, which only current members possess.

This split is deliberate. The identity key never rotates; the epoch key rotates on every membership change. Both invariants matter: a stable identity key means users can refer to `@team-design` and have the reference persist across rotations; rotating epoch keys means a removed member can no longer read new messages even though they still hold the old epoch key.

### 1.3 Audience declaration event

A new addressable kind is introduced for the audience identity:

| Kind | Object | Replaceability |
|---|---|---|
| **30520** | `fa:Audience` | Addressable by `d` (audience slug) |

The declaration is signed by `aud_id` itself (the audience's identity keypair acts as a Nostr publisher). Its content carries the human-facing name, description, current epoch number, and the current `aud_epoch_n` pubkey. Its tags include one `p` per current member pubkey — this is the **public roster**, used for discoverability, *not* for access control. (Access is gated by who actually holds `aud_epoch_n` via key-grants — see below.)

Sketch:

```json
{
  "kind": 30520,
  "tags": [
    ["d", "team-design"],
    ["fa:context", "https://4a4.ai/ns/v0"],
    ["alt", "Audience: team-design (4 members, epoch 7)"],
    ["fa:epoch", "7"],
    ["fa:epoch-pubkey", "<hex pubkey of aud_epoch_7>"],
    ["p", "<member-1 pubkey>"],
    ["p", "<member-2 pubkey>"],
    ["p", "<member-3 pubkey>"],
    ["p", "<member-4 pubkey>"]
  ],
  "content": "{\"@context\":\"https://4a4.ai/ns/v0\",\"@type\":\"Audience\",\"name\":\"team-design\",\"description\":\"Design notes shared with Allison.\",\"epoch\":7}"
}
```

Because it is addressable, replacing it (e.g. when rotating to epoch 8) supersedes the previous declaration without rewriting history.

### 1.4 Key-grants

A **key-grant** is the wire mechanism by which one member shares the current epoch private key with another. It uses the encrypted-variant kind block reserved in [`kind-assignments.md`](./kind-assignments.md):

| Kind | Object | Encrypted to | Replaceability |
|---|---|---|---|
| **30521** | `fa:KeyGrant` | recipient member pubkey | Addressable by `d = audience-slug:epoch:recipient` |

The grant is published by an existing member (or, for the first grant in an audience, by the founder). Its `content` is a NIP-44 ciphertext, encrypted from the granter's identity key to the recipient member's identity key, carrying the epoch private key in the clear plaintext. Tags include `p` for the recipient (so they can filter), `a` for the audience declaration, and `fa:epoch` for the epoch number.

```json
{
  "kind": 30521,
  "tags": [
    ["d", "team-design:7:<recipient-pubkey>"],
    ["fa:context", "https://4a4.ai/ns/v0"],
    ["alt", "KeyGrant: team-design epoch 7"],
    ["a", "30520:<aud_id pubkey>:team-design"],
    ["fa:epoch", "7"],
    ["p", "<recipient pubkey>"]
  ],
  "content": "<NIP-44 ciphertext containing aud_epoch_7 private key>"
}
```

A member subscribes to `kinds:[30521], #p:[my pubkey]` and decrypts whatever lands. From that ciphertext they recover `aud_epoch_n_priv` and store it locally indexed by `(audience, epoch)`.

### 1.5 Addressing a message to an audience

To publish a 4A object (Observation, Claim, Entity, Relation) to an audience, the author:

1. Looks up the current `aud_epoch_n` pubkey from the audience declaration.
2. Builds the normal v0 4A payload (JSON-LD, BLAKE3, all the rest — see [`SPEC.md` §Payload format](./SPEC.md)).
3. NIP-44-encrypts the serialized payload to `aud_epoch_n` pubkey.
4. Publishes as the **encrypted-variant kind** (30510 for Observation, 30511 for Claim, 30512 for Entity, 30513 for Relation, 30514 for an audience-scoped Commons), with tags:
   - `a` pointing at the audience declaration (`30520:<aud_id pubkey>:<slug>`)
   - `fa:epoch` carrying the epoch number it was encrypted to
   - `p` per each current member pubkey, to allow per-recipient gift-wrapping (next bullet)
5. Wraps the resulting event with [NIP-17 gift-wrap](https://github.com/nostr-protocol/nips/blob/master/17.md) (kind 1059) **once per recipient member**, so that the public envelope reveals only that *some* sealed event was sent to *some* recipient, not the audience name, the epoch, or the membership graph.

The gift-wrap step is the key metadata-privacy move. Without it, a relay operator could link members of an audience by watching `#a` filters; with it, only the recipient learns which audience the inner event belongs to. Gift-wrap is more expensive on the publish side (one wrap per member) but is what makes the model usable for sensitive groups. v0.5 specifies gift-wrap as a **MUST** for audience-addressed events; the fan-out cost is acceptable at small audience sizes (typical: 2–20 members).

### 1.6 Per-epoch rotation

Rotation is triggered by:

- **Member added.** New epoch generated, new key-grants issued to all members including the new one.
- **Member removed.** New epoch generated, new key-grants issued to all *remaining* members. The removed member's identity key is no longer a recipient.
- **Periodic refresh** (SHOULD). Audiences with sensitive content rotate on a fixed cadence (default 90 days) regardless of membership changes, to bound the blast radius of an undetected key compromise.

After rotation, old encrypted events remain readable with the old `aud_epoch_n_priv` (which existing members still have); new events are encrypted to `aud_epoch_n+1`, which removed members do not have. This gives **forward secrecy on membership changes** without requiring the [NIP-104 / MLS-on-Nostr](https://github.com/nostr-protocol/nips/blob/master/104.md) machinery, which is not yet stable. When NIP-104 stabilizes, audiences will migrate; the wire-level shape (encrypted-variant kinds + audience declaration) is designed so that swapping the encryption layer underneath is backward-compatible at the JSON-LD level.

### 1.7 What audiences deliberately do not provide

- **Forward secrecy within an epoch.** If the current epoch key leaks, every message in that epoch is exposed. Rotation bounds the window.
- **Post-compromise security.** A compromised member's identity key allows reading every key-grant that was ever encrypted to them. Mitigation: identity keys rotate via the OAuth-derivation counter (see §2.3), and members SHOULD re-issue grants under the new identity.
- **Hidden membership at the audience-level.** The public roster (`p` tags on the Audience declaration) is published in the clear. Audiences that need hidden membership SHOULD omit the public roster and rely solely on key-grants, accepting a discoverability cost.

These limits are honest acknowledgments, not bugs. They pin the v0.5 audience model at the right altitude: better than public-only Nostr, weaker than MLS, useful enough to ship two-person and small-team scenarios today.

## 2. Key management

### 2.1 What v0.5 commits to

v0.5 fixes the **deterministic OAuth-derived key** scheme already shipped in Phase 2 ([`ARCHITECTURE.md` §Identity model](./ARCHITECTURE.md)) as the default user identity for the hosted gateway. Restated as a v0.5 contract:

```
oauth_id_string = oauth_provider + ":" + oauth_user_id
seed_bytes      = AWS_KMS.GenerateMac(KeyId=KMS_DERIVATION_KEY_ID,
                                      Message=oauth_id_string,
                                      MacAlgorithm="HMAC_SHA_256")
nostr_priv      = clamp_to_secp256k1(seed_bytes)
nostr_pub       = secp256k1_pubkey(nostr_priv)
```

The KMS HMAC key is non-extractable. Seed bytes are returned per signing request, used in-memory for the duration of one Worker invocation, and discarded. **No key material persists anywhere.** Re-authenticating to GitHub re-derives the same key.

### 2.2 No vault — the explicit choice

We do **not** run a key vault. Every alternative considered (per-user encrypted blob in DynamoDB, AWS Secrets Manager record per user, Cloudflare KV with Worker-side encryption, HashiCorp Vault, etc.) was rejected for the same reason: a vault is a database, a database is a thing that can be breached, and a breach of *that* database leaks every user's private key at once. The deterministic-derivation approach replaces "secret-at-rest" with "secret-derivable-on-demand-from-OAuth." The KMS HMAC root is the only persistent secret, it lives in an HSM, and it never leaves.

The cost of this choice is the property described in the next subsection.

### 2.3 The lose-OAuth-lose-access tradeoff

**If a user loses access to their GitHub OAuth account, they lose access to their 4A identity.** There is no recovery path through us. We do not hold a copy of their private key; we cannot derive it without their OAuth identity proving who they are.

This is real and it should be stated to users plainly at sign-up. Three things mitigate it:

1. **`GET /me/export` is offered up front, not buried.** The first time a user lands on the hosted gateway dashboard, the flow surfaces "Export your private key" alongside "Continue." Users who export get a copy of `nostr_priv` they can self-custody (paper, password manager, hardware wallet). Users who skip are explicitly accepting the OAuth-loss risk.
2. **The OAuth provider itself has account recovery.** GitHub's recovery flow (recovery codes, phone, etc.) is the practical recovery mechanism for most users. We are leaning on the OAuth provider to be the keystore, which is a reasonable bet for the engineer audience the OSS-commons wedge targets.
3. **The NIP-46 bunker escape hatch (next subsection)** is available for users who want a second, non-derivable identity.

Rotation in the deterministic scheme is supported by adding a counter to the derivation input:

```
oauth_id_string = oauth_provider + ":" + oauth_user_id + ":" + rotation_counter
```

Where the counter lives is a deferred decision (JWT claim, user-published NIP-32 self-label, or tiny KV row). v0.5 does not implement counter-rotation; it pins the contract as "your OAuth identity is your 4A pubkey, until you explicitly rotate."

### 2.4 The NIP-46 power-user escape hatch

Any user who does not want a custodial key — either philosophically, or because they already have a Nostr identity, or because they want post-compromise control over rotation — provides a [NIP-46](https://github.com/nostr-protocol/nips/blob/master/46.md) bunker URI in their account settings. From that point forward, the gateway forwards signing requests to the bunker; the bunker holds the key; the gateway never sees plaintext private material.

NIP-46 is the same primitive that nsec.app and self-hosted bunker software already implement. The hosted gateway adds NIP-46 as an alternate identity path in v0.5 (see [`ARCHITECTURE.md` §Phase 2](./ARCHITECTURE.md)) — it is not a bespoke fork.

This split is intentional. The custodial path is the **default** because most users won't (and shouldn't have to) think about key custody to publish their first 4A object. The bunker path is the **escape hatch** for users who have stronger requirements. Both produce indistinguishable signed events at the wire.

### 2.5 Private mode for everyone — capability-based delegation

Audiences are **capability-based**: anyone holding the current audience epoch key can decrypt content addressed to it. Identity is decoupled from access. A user joining an audience receives the epoch key via a key-grant; from that point, decryption depends only on possession of the key, not on which surface (local client, hosted gateway, browser) does the decrypting.

This means **private mode is available to every user, on every surface**. The trust posture differs by surface, not the capability:

- **Local-client surfaces** (Sonata plugin, browser extension, `4a` CLI). The audience epoch key lives in the user's local keystore. Decryption happens on the user's machine. The gateway transports ciphertext only.
- **Hosted-gateway custodial surfaces** (ChatGPT Custom GPT, Claude.ai connector, anything that talks to `api.4a4.ai` over OAuth). The user delegates decryption to the gateway under the same trust posture that already governs signing in Phase 2. The gateway derives the user's identity private key on demand via KMS, uses it to NIP-17 unwrap and to decrypt the relevant key-grant, then uses the recovered audience epoch key to decrypt the content. All key material is transient — nothing persists, nothing is stored in any vault.

The per-read cost the gateway pays for capability-based delegation:

1. One `KMS.GenerateMac` call to derive the user's identity private key (~50 ms, ~$0.0000003).
2. One NIP-17 unwrap (ECDH + symmetric decrypt, sub-millisecond).
3. One NIP-44 key-grant decrypt to recover the audience epoch private key. The most-recent key-grant for `(user, audience, epoch)` is looked up in the relay-pool Durable Object's existing index of kind 30521 events; no new infrastructure required.
4. One NIP-44 content decrypt with the audience epoch key.
5. JSON-LD parse and return. All transient key material discarded at the end of the request.

This is the same in-memory window the gateway already uses for signing — the worker holds key material for the duration of one request and discards it. **No vault, no per-user keystore, no audience-key store.**

The threat model this commits to honestly:

- **Compromise of one local client** = audience access for that one user, until rotation.
- **Compromise of the hosted gateway** = the ability to derive any custodial user's identity private key (already true today for signing) AND the ability to decrypt audiences for any custodial user who has delegated. The blast radius covers the cohort of custodial users who have ever delegated decryption — in practice the entire ChatGPT / Claude.ai user base. This is strictly larger than today's signing-only exposure.
- **Compromise of the KMS HMAC root** would allow offline derivation of every custodial user's identity key, and through that any audience key those users have ever held. KMS HMAC keys are non-extractable by definition; this is "AWS itself is rooted" territory.

We accept the gateway-compromise blast radius as the cost of letting cloud-agent users participate in private mode. A user who wants stronger isolation runs a local client; that path is always open and the audience key follows the user, not the surface — a user can delegate to the gateway today and switch to sole custody on their laptop tomorrow without re-keying.

In practice this looks like:

| Surface | Public publishing | Private publishing | Private reading |
|---|---|---|---|
| ChatGPT Custom GPT | gateway-signed | gateway-signed (custodial delegation) | gateway-decrypted (custodial delegation) |
| Claude.ai connector | gateway-signed | gateway-signed (custodial delegation) | gateway-decrypted (custodial delegation) |
| Sonata plugin (local) | client-signed | client-signed | client-decrypted |
| Browser extension | client-signed | client-signed | client-decrypted |
| Local `4a` CLI | client-signed | client-signed | client-decrypted |

The v0.5 stance: **everyone can use private mode.** The gateway acts as a delegated decryption surface for custodial users with the same trust posture it already uses for signing. Local clients remain the recommended path for users who want neither their identity key nor their audience keys to enter the hosted gateway.

## 3. Invite flow

### 3.1 Design constraint: one paste field

The interaction we want is one text input, labeled "Invite someone." The user types or pastes:

- A 4A handle (`alice@4a4.ai`, `bob@enginable.com`)
- An email address (`allison@enginable.com`)
- An npub (`npub1...`)
- Nothing — leave blank, hit "Generate link"

The system interprets the input and produces the right next step. There is no separate "invite by email" vs "invite by npub" tab. The single-paste constraint is the UX commitment of v0.5.

### 3.2 The off-4A primitive: claimable URL

Underneath every invite flow is the same wire primitive: a **claimable URL**.

```
4a://invite/<audience-slug>/<epoch>?k=<invite_priv>
```

Where:

- `<audience-slug>` is the audience's `d` tag (e.g. `team-design`).
- `<epoch>` is the epoch the invite is scoped to.
- `<invite_priv>` is a freshly-generated, single-use secp256k1 private key whose corresponding pubkey is added to the audience's roster as a "pending" member.

When the invitee opens the URL in their 4A client (or web claim page on `claim.4a4.ai`), the client:

1. Parses the URL.
2. Uses `invite_priv` to authenticate as the pending member.
3. Posts a "claim" event to the relay: a small signed message asking the inviter to issue a real key-grant to the invitee's *actual* identity pubkey (whichever they pick — custodial or bunker).
4. The inviter's client (or the inviter's hosted-gateway daemon) sees the claim, generates a key-grant from `aud_epoch_n_priv` to the invitee's identity pubkey, publishes it (kind 30521), and rotates the audience to a new epoch (so the throwaway `invite_priv` is no longer a current member).
5. The invitee receives the key-grant on their normal subscription and is now a full member.

This decouples *delivery* of the invite from *redemption*. The 4A protocol does not deliver invite URLs — that's the inviter's problem (paste into iMessage, email, Slack, walk it across the room on a USB stick). The protocol only specifies what to do when one shows up.

### 3.3 The single-paste field, dispatched

The four cases the paste field handles:

| Input | Resolution |
|---|---|
| `alice@4a4.ai` (a 4A handle) | Resolve to npub via NIP-05 directory (§4). Issue a key-grant directly to `alice`'s identity pubkey, no claim flow needed. |
| `alice@enginable.com` (a 4A handle on a federated host) | Same as above — NIP-05 lookup against `enginable.com/.well-known/nostr.json`. |
| `allison@enginable.com` (an email address that does **not** resolve as NIP-05) | Generate a claimable URL. Inviter is shown the URL and a one-click "Send by email via my email client" button. Delivery is the inviter's problem. |
| `npub1...` (a raw npub) | Issue a key-grant directly. No claim flow needed. The inviter is responsible for verifying the npub belongs to the right person. |
| (blank) + "Generate link" | Same as the email case minus the prompt to send. Inviter pastes the URL wherever they like. |

The handle and npub paths are direct: identity is already known, we can issue the grant. The email and link paths are claim-mediated: we don't know the invitee's identity yet, so we hand out a one-shot key that they trade in for one.

### 3.4 TTL rotation and single-use

Two SHOULDs that protect the claimable URL primitive:

- **TTL rotation (SHOULD).** Pending invites SHOULD carry a default expiration of 7 days. The audience's epoch declaration SHOULD list pending invite pubkeys with their expiration; clients SHOULD purge expired pending pubkeys at rotation time. This bounds the window during which a leaked URL is useful.
- **Single-use claim watch (SHOULD).** The inviter's client SHOULD subscribe to the relay for claim events from the pending pubkey. On receiving the *first* claim, it SHOULD issue the key-grant and immediately rotate the audience to a new epoch (which removes the pending pubkey). Subsequent claim attempts on the same URL fail because the pending pubkey is no longer in the roster.

Both are SHOULDs not MUSTs because some surfaces (web-only invitee, no inviter client running) won't be able to honor them strictly. The single-use property degrades to "the second claim attempt loses a race" rather than "the second claim attempt is rejected by the protocol."

### 3.5 What this is *not*

- Not a magic link in the sense of "click and you're authenticated." The invitee still needs to authenticate as themselves (custodial OAuth or bunker) and present an identity pubkey. The claimable URL only carries the audience-side credential.
- Not a delivery mechanism. We do not host an email-sending service for invites. If the inviter pastes an email address, we generate a URL and a `mailto:` link; the user's actual email client sends.
- Not a permanent capability. Default TTL is 7 days; epoch rotation invalidates pending invites that haven't claimed.

## 4. Directory

### 4.1 Three directory tiers, one shipping

The community has discussed three directory shapes for translating a human-readable handle into a Nostr pubkey:

- **D1 — Federated lookup.** A `name@host` handle resolves via a static-file convention served by `host`. Existing precedent: [NIP-05](https://github.com/nostr-protocol/nips/blob/master/05.md), which uses `https://<host>/.well-known/nostr.json?name=<name>`.
- **D2 — Pure relay query.** Handles are stored as relay events; resolution is a relay subscription. No host required.
- **D3 — Petnames.** Handles are local nicknames in each client's address book; no global resolution.

v0.5 ships **D1 as the primary lookup convention** and **D3 layered in clients**. **D2 is explicitly skipped** for v0.

The reasoning: D1 piggybacks on existing NIP-05 infrastructure, requires no new server software (just a JSON file at a `.well-known` path), and gives users handles that look like email addresses — which they understand. D2 is appealing on the "no DNS" axis but pushes a hard discovery problem onto the relay layer (which relays do you query? all of them? how do you authenticate the response? what about handle squatting?) that we are not equipped to answer in v0. D3 is what every client should do anyway; it's not a substitute for a global lookup but it's the right local UX.

### 4.2 D1 — federated NIP-05

The convention: a 4A handle is `<name>@<host>`. To resolve it, the client fetches:

```
GET https://<host>/.well-known/nostr.json?name=<name>
```

Expected response (the standard NIP-05 shape, plus a 4A extension key):

```json
{
  "names": {
    "alice": "<hex pubkey>"
  },
  "relays": {
    "<hex pubkey>": ["wss://relay.damus.io", "wss://nos.lol"]
  },
  "fa": {
    "<hex pubkey>": {
      "audiences": ["team-design", "alice-public"],
      "context": "https://4a4.ai/ns/v0"
    }
  }
}
```

The `names` and `relays` keys are stock NIP-05. The `fa` key is a 4A-specific extension that lists known audiences this user publishes to (so a client resolving `alice@4a4.ai` can offer "subscribe to alice's `alice-public` audience" without an extra round trip) and pins the context version for compatibility checks. The `fa` extension is OPTIONAL; clients MUST function with hosts that publish only the standard NIP-05 fields.

### 4.3 4a4.ai is one host among many

This is important enough to call out explicitly: **`4a4.ai` will run as one NIP-05 host (handles like `alice@4a4.ai`), but the spec MUST treat it as one host among many.** There is no privileged registry. There is no "global 4A directory." Anyone who controls a domain can publish a `.well-known/nostr.json` and own the namespace under that domain.

Consequences of this decision:

- Enterprises can self-host their employee directory: `alice@enginable.com` resolves on `enginable.com` against `enginable.com`'s own NIP-05 file.
- Open-source projects can self-host commons handles: `next.js@vercel.com` could be the official Next.js commons publisher, and `vercel.com` is the root of trust.
- A 4A client MUST NOT hardcode `4a4.ai` as a fallback resolver. It is not the network. It is one node.

`4a4.ai`'s role is to make sign-up easy for the no-domain user — they get a free handle on a host we run. They can leave whenever they like by publishing under their own domain.

### 4.4 D3 — petnames in clients

On top of D1, every client is encouraged to maintain a local petname store: "I've talked to `alice@4a4.ai` enough times that in my UI she's just `alice`." Petnames are local, mutable, and never published to relays. This is the [Zooko triangle](https://en.wikipedia.org/wiki/Zooko%27s_triangle) move: pick two of (decentralized, secure, human-meaningful), get the third by composition. D1 gives us decentralized + secure handles; D3 gives human-meaningful local renderings.

The spec's contribution to D3 is: a recommended JSON shape for petname entries (`{"alias": "alice", "pubkey": "<hex>", "verified_via": "alice@4a4.ai", "first_seen": "<timestamp>"}`), a recommendation that clients warn the user when a petname's underlying pubkey changes (handle hijack detection), and a recommendation that petnames sync via the same client's own backup mechanism, not through Nostr.

### 4.5 What we publish and what we don't

- **We publish a lookup convention.** A short document specifying the `.well-known/nostr.json` shape with the `fa` extension, plus the petname shape recommendation.
- **We do not publish a directory service.** There is no central API for "look up any 4A user." There is no curl-able registry of all 4A handles. The convention is the spec; resolution is HTTP.

This is the same posture 4A takes generally: convention, not service. We are doing for handles what the public-mode spec does for events.

## 5. Worked example — Evan creates "team-design", invites Allison by email, both publish and read

This trace walks the entire v0.5 stack from cold start through the first round-trip. The Sonata client is assumed on Evan's machine; Allison is invited by email and chooses to claim through `claim.4a4.ai` in her browser.

### 5.1 Evan signs in

Evan visits `https://4a4.ai`, clicks **Continue with GitHub**, and authorizes the OAuth app. The hosted gateway:

1. Receives the OAuth callback at `https://api.4a4.ai/auth/github/callback`.
2. Reads the GitHub user id (`12345`) from the OAuth profile response.
3. Constructs the derivation input: `oauth_id_string = "github:12345"`.
4. Calls `AWS_KMS.GenerateMac(KeyId=KMS_DERIVATION_KEY_ID, Message="github:12345", MacAlgorithm="HMAC_SHA_256")`.
5. Clamps the returned 32 bytes to a valid secp256k1 scalar; that is `nostr_priv_evan`.
6. Computes `nostr_pub_evan = secp256k1_pubkey(nostr_priv_evan)`.
7. Mints a 24-hour HS256 JWT carrying `{sub: "github:12345", pub: "<nostr_pub_evan hex>"}` and sets it as a session cookie.
8. Discards `nostr_priv_evan` from memory.

Evan now has a 4A identity. He has not stored anything; his pubkey is whatever `KMS.GenerateMac` deterministically derives from `"github:12345"`. The first-time UI surfaces a "Save your private key" panel; Evan clicks **Show**, the gateway re-derives, displays the nsec form, and Evan saves it to his password manager. (Optional but encouraged.)

### 5.2 Evan creates the audience

Evan opens the Sonata plugin (his local 4A client). The plugin authenticates to the gateway with the JWT cookie and asks the gateway to publish a public-mode test event ("hello, 4A"). That works — Evan's identity is wired up.

He clicks **New audience** and types `team-design` as the slug, "Design notes shared with Allison" as the description. The plugin:

1. Generates a fresh secp256k1 keypair locally: `aud_id_priv` and `aud_id_pub`.
2. Generates a second fresh keypair for the first epoch: `aud_epoch_1_priv` and `aud_epoch_1_pub`.
3. Stores both privately in the plugin's local keystore, indexed by audience slug.
4. Constructs the audience declaration event (kind 30520):
   - `d = "team-design"`
   - `fa:epoch = "1"`
   - `fa:epoch-pubkey = <hex aud_epoch_1_pub>`
   - `p = [<hex nostr_pub_evan>]` (only Evan, for now)
   - signed by `aud_id_priv`.
5. Publishes the declaration to Evan's configured relay set (`relay.damus.io`, `nos.lol`, `nostr.wine`).
6. Issues the founding key-grant: a kind-30521 event with `d = "team-design:1:<nostr_pub_evan>"`, content = `NIP-44_encrypt(plaintext = aud_epoch_1_priv, sender = aud_id_priv, recipient = nostr_pub_evan)`. Publishes.

Evan now has an audience with one member (himself) on epoch 1. He can already address messages to it; they just only land in his own inbox.

### 5.3 Evan invites Allison

In the audience's settings, Evan clicks **Invite** and pastes `allison@enginable.com` into the single paste field. The plugin:

1. Tries NIP-05 resolution: `GET https://enginable.com/.well-known/nostr.json?name=allison`. Returns 404 — Allison's company doesn't run a NIP-05 directory.
2. Falls back to the claimable-URL flow.
3. Generates `invite_priv` and `invite_pub` (a one-shot keypair).
4. Adds `invite_pub` to the audience's roster as a "pending" entry, valid for 7 days, by replacing the audience declaration with a new version that carries one extra `p` tag and a corresponding `fa:pending` tag with the expiration.
5. Builds the URL: `4a://invite/team-design/1?k=<base32 invite_priv>` and a parallel HTTPS form: `https://claim.4a4.ai/invite/team-design/1?k=<base32 invite_priv>`.
6. Pops a system dialog: "Send Allison this link." Evan clicks **Send by email**, which opens his Mail app with the URL pre-filled, addressed to `allison@enginable.com`. Evan sends.
7. The plugin starts a relay subscription `kinds:[30501-equivalent claim event], #p:[invite_pub]` to watch for Allison's claim.

Delivery is now Evan's problem. The protocol has handed him a URL and washed its hands.

### 5.4 Allison claims

Allison receives the email, clicks the HTTPS link. `claim.4a4.ai` (a thin static page served from Cloudflare Pages) loads in her browser. The page:

1. Parses the URL: extracts `audience = "team-design"`, `epoch = 1`, `invite_priv`.
2. Asks Allison to authenticate. She has no 4A identity; she clicks **Continue with GitHub**, runs through the OAuth flow, and the hosted gateway derives `nostr_pub_allison` from `"github:67890"`.
3. The page now has both `invite_priv` (from the URL) and `nostr_pub_allison` (from her fresh OAuth). It builds a "claim" event signed by `invite_priv`, with content "I am `nostr_pub_allison`, please grant me", and publishes it to a configured relay.

Evan's plugin sees the claim event on its subscription. It:

1. Verifies the claim is signed by the pending `invite_pub`.
2. Generates a new epoch (`aud_epoch_2_priv`, `aud_epoch_2_pub`).
3. Issues two key-grants: one to `nostr_pub_evan`, one to `nostr_pub_allison`. Both grants encrypt `aud_epoch_2_priv`.
4. Replaces the audience declaration: `epoch = 2`, `fa:epoch-pubkey = <aud_epoch_2_pub>`, `p = [<nostr_pub_evan>, <nostr_pub_allison>]`. Pending entry removed.
5. Notifies Evan locally: "Allison joined team-design."

Allison's browser, on its own subscription `kinds:[30521], #p:[nostr_pub_allison]`, receives the key-grant. The claim page decrypts it in the browser (this specific bootstrap step happens client-side because Allison is in a web claim flow; ongoing reads can use whichever surface she chooses — see §2.5), recovers `aud_epoch_2_priv`, and offers her a choice: keep the audience key local (browser IndexedDB / Sonata plugin / `4a` CLI) or delegate it to the hosted gateway so she can read this audience from ChatGPT or Claude.ai later. The page redirects Allison to a "You're in" view that shows the audience's existing messages (none yet) and lists the available surfaces.

### 5.5 First message round-trip

Evan, back in his Sonata plugin, drafts an Observation: "Saw a slick CSS reset that drops `margin: 0` on `body`." The plugin:

1. Builds the v0 4A Observation payload (full JSON-LD per [`SPEC.md` §Observation](./SPEC.md), with BLAKE3 tag, `alt` tag, the works).
2. Looks up the audience's current epoch: 2, with `aud_epoch_2_pub`.
3. NIP-44-encrypts the serialized Observation to `aud_epoch_2_pub`. The result is the inner ciphertext.
4. Wraps the inner ciphertext as a kind-30510 event (encrypted-Observation variant), signed by `nostr_pub_evan`, tagged with `a = 30520:<aud_id_pub>:team-design`, `fa:epoch = "2"`, and `p` for each current member.
5. Gift-wraps the kind-30510 event twice — once per member — using NIP-17. Each wrap is a kind-1059 outer event encrypted to one recipient. The outer event reveals only that some sealed payload was sent to that recipient.
6. Publishes both gift-wraps to the relay set.

Allison's client (whichever surface — assume she installed the Sonata plugin) holds a subscription `kinds:[1059], #p:[nostr_pub_allison]`. She receives the gift-wrap addressed to her, NIP-17-unwraps to recover the inner kind-30510 event, looks up `aud_epoch_2_priv` from her local store, NIP-44-decrypts the content, parses the JSON-LD against the 4A context (`https://4a4.ai/ns/v0`), and renders: "Evan, in team-design: Saw a slick CSS reset that drops margin: 0 on body."

Allison replies with her own Observation. Same flow in reverse: build payload → encrypt to `aud_epoch_2_pub` → kind 30511 (encrypted Observation) → gift-wrap per member → publish. Evan's plugin receives, unwraps, decrypts, renders.

The total moving parts on the wire, end-to-end, for the first message:

| # | Event | From | To | Kind | Encryption |
|---|---|---|---|---|---|
| 1 | Audience declaration v1 | `aud_id` | public | 30520 | none |
| 2 | KeyGrant epoch 1 → Evan | `aud_id` | Evan | 30521 | NIP-44 |
| 3 | Audience declaration v1' (with pending invite) | `aud_id` | public | 30520 | none |
| 4 | Claim from invite_pub | `invite_priv` | public | (custom) | none |
| 5 | Audience declaration v2 (Allison member, epoch 2) | `aud_id` | public | 30520 | none |
| 6 | KeyGrant epoch 2 → Evan | `aud_id` | Evan | 30521 | NIP-44 |
| 7 | KeyGrant epoch 2 → Allison | `aud_id` | Allison | 30521 | NIP-44 |
| 8 | Evan's Observation (inner) | Evan | audience | 30510 | NIP-44 to aud_epoch_2 |
| 9 | Gift-wrap of (8) → Evan | Evan | Evan | 1059 | NIP-17 |
| 10 | Gift-wrap of (8) → Allison | Evan | Allison | 1059 | NIP-17 |

Ten relay round-trips, all built from existing Nostr primitives, no new infrastructure. That is the v0.5 commitment.

### 5.6 What the example demonstrates

- **OAuth-derived identity is invisible to the wire.** Both Evan's and Allison's pubkeys are derived from `KMS.GenerateMac("github:<id>")`. Nothing in the published events signals "this user is custodial." The events look like any other Nostr signed events.
- **Audiences are layered on top of public 4A.** The encrypted-variant kinds (30510–30514) are 1:1 with the public kinds; the audience declaration (30520) and key-grants (30521) are addressable events that can be supplied by any relay.
- **Invites do not require email infrastructure.** The protocol generates a URL; the user's existing email client sends it. We host nothing on the delivery path.
- **Directory resolution is best-effort.** The example failed at NIP-05 lookup (`enginable.com` doesn't run one) and degraded to the claim-link flow. A successful NIP-05 lookup would have skipped step 4 entirely.
- **Decryption is capability-based, not surface-bound.** Anyone holding the audience epoch key can read; the choice of surface (local client, hosted gateway under custodial delegation, browser claim page) only changes the trust posture, not the capability. In this example Allison happens to be on the Sonata plugin; she could equally be on the Claude.ai connector or the ChatGPT GPT and read the same audience via gateway-delegated decryption. See §2.5.

## Decisions locked by this document

For posterity (and so the next session doesn't relitigate):

1. **Audience model = identity key + per-epoch keypair + key-grants + NIP-44 + NIP-17 gift-wrap.** Forward secrecy on membership changes via epoch rotation; periodic rotation as a SHOULD; NIP-104 migration deferred until that NIP stabilizes.
2. **New kinds:** 30520 (`fa:Audience`), 30521 (`fa:KeyGrant`). Encrypted variants 30510–30514 (already reserved in [`kind-assignments.md`](./kind-assignments.md)) carry audience-addressed payloads.
3. **Key management = deterministic OAuth-derived (Phase 2 shipped).** No vault. Lose-OAuth-lose-access stated honestly. Export endpoint surfaced at first sign-in. NIP-46 bunker as power-user escape hatch.
4. **Private mode is capability-based and available on every surface.** Anyone holding an audience's epoch key decrypts; identity is decoupled from access. Custodial users on cloud surfaces (ChatGPT GPT, Claude.ai connector) participate via gateway-delegated decryption — the gateway derives the user's identity key on demand via KMS, decrypts the relevant key-grant, and uses the recovered audience epoch key to decrypt content, all in the same transient in-memory window already used for signing. No vault, no persisted audience-key store. Local clients (Sonata plugin, browser extension, `4a` CLI) remain the recommended path for users who want stronger isolation; the audience key follows the user, not the surface, so a user can move between modes without re-keying.
5. **Invite UX = single paste field.** Accepts handle, email, npub, or blank. Off-4A wire primitive is a claimable URL. Inviter delivers. TTL 7 days SHOULD; single-use claim watch SHOULD; rotation on first claim SHOULD.
6. **Directory = D1 federated NIP-05 + D3 client-side petnames.** D2 (pure relay query) skipped for v0.5. `.well-known/nostr.json` extended with an OPTIONAL `fa` key. `4a4.ai` is one NIP-05 host among many; spec MUST NOT privilege it.
7. **No central directory service.** We publish a lookup convention, not a server.

## Out of scope for v0.5

- NIP-104 / MLS-on-Nostr group encryption. Reserved for a future revision once the upstream NIP stabilizes.
- Counter-based rotation of OAuth-derived keys. Mechanism sketched in §2.3; v0.5 does not implement.
- Public NIP-85 score aggregator. Credibility primitives (labels, sponsorships, stamps) ship in v0; the aggregator that *consumes* them is Phase 2.5+.
- Cross-audience message routing (one event → multiple audiences). Audiences are independent recipient sets in v0.5.
- Pinned-version JSON-LD context (`https://4a4.ai/ns/v0#2026-04-24`). Mentioned in [`SPEC.md`](./SPEC.md); not formalized here.
